#### 文件 I/O 

> **带缓冲 I/O 是建立在系统调用之上的性能优化层，两者是不同层次的工具，根据场景选择。**  
> 普通文件 → 用标准 I/O；网络/设备/实时 → 用系统调用 I/O。

| 类型 | 实现者 | 典型函数 | 特点 |
|------|--------|----------|------|
| **带缓冲 I/O**（标准 I/O） | C 标准库 | `fopen`, `fread`, `fwrite`, `printf`, `fprintf`, `fgets` | 用户态缓冲区，减少系统调用 |
| **不带缓冲 I/O**（系统调用 I/O） | 操作系统内核 | `open`, `read`, `write`, `close`, `lseek` | 直接通过文件描述符操作内核 |

> 注：“不带缓冲”仅指**用户态无自动缓冲**，内核依然有页缓存等机制。

1. C 标准库的三种缓冲模式

| 模式 | 行为 | 典型对象 |
|------|------|----------|
| **全缓冲** | 填满缓冲区才进行实际 I/O | 普通磁盘文件 |
| **行缓冲** | 遇到 `\n` 或缓冲区满才输出 | 终端（stdout） |
| **无缓冲** | 立即输出 | stderr（错误信息） |

2. 核心区别对比表

| 维度 | 带缓冲 | 不带缓冲 |
|------|--------|----------|
| **性能（小读写）** | 高 | 极低（大量系统调用） |
| **实时性** | 低（有延迟） | 高（立即） |
| **系统调用次数** | 少 | 多 |
| **原子性** | 多线程需加锁 | 小数据量下可能有原子性 |
| **进程 fork 后** | 子进程复制缓冲区，可能重复输出 | 无用户缓冲区，安全 |
| **崩溃数据安全** | 未刷新缓冲区数据丢失 | 已 write 的数据进入内核更安全 |
| **格式化输出** | 方便（printf 系列） | 不便（需自己格式化） |
| **适用场景** | 普通文件、文本处理 | 网络、设备、实时、非阻塞 I/O |

3. 调用关系（重要！）

> 带缓冲 I/O 最终一定会调用不带缓冲 I/O（系统调用）。

```
fwrite() → 写入用户态缓冲区 → (缓冲区满/fflush) → write() 系统调用 → 内核
```

- 用户态缓冲区只是减少系统调用次数，无法完全消除。
- 最终数据必须通过内核才能落盘或发送到设备。

5. 性能示例

写入 1 亿字节，每次 1 字节：

- **不带缓冲**：1 亿次 `write` 系统调用 → 极慢
- **带缓冲**（默认 8KB）：约 12207 次 `write` 系统调用 → 快数千倍

尽管带缓冲 I/O 性能好，但以下场景**必须或不适合**使用标准 I/O 缓冲：

  - **实时/交互设备**（串口、socket）：希望数据立即发送，而不是等缓冲区满。
  - **非阻塞/异步 I/O**：`read`/`write` 可与 `select`/`epoll` 配合，标准 I/O 的缓冲会破坏非阻塞语义。
  - **大块数据传输**：直接 `read(fd, buf, 1MB)`，避免双缓冲（用户自己已有缓冲区）。
  - **原子追加**：多进程写同一文件，用 `O_APPEND` + `write` 保证原子性，`fwrite` 可能被打断。
  - **避免内存拷贝**：带缓冲 I/O 多一次用户态内存拷贝。

6. 选择指南

| 场景 | 推荐 |
|------|------|
| 读写普通文件（大量小数据） | 带缓冲 |
| 文本处理、格式化输出 | 带缓冲（`fprintf`） |
| 网络编程（TCP/UDP） | 不带缓冲（`send`/`recv`） |
| 串口通信 | 不带缓冲 |
| 高并发 + 非阻塞 I/O | 不带缓冲 |
| 需要精细控制缓存（如 Direct I/O） | 不带缓冲 |
| 跨平台（Windows/Linux/嵌入式） | 带缓冲（标准 C 可移植） |

7. 常见误区澄清

  - **“不带缓冲 I/O 更快”** → 错。多数情况下带缓冲更快，除非每次读写非常大。
  - **“`printf` 立即输出”** → 不一定。输出到终端是行缓冲，需要 `\n` 或 `fflush`。
  - **“`fwrite` 后数据已落盘”** → 错。仅写入用户态缓冲区，需要 `fflush` + `fsync` 才能落盘。
  - **“带缓冲 I/O 与不带缓冲完全无关”** → 错。带缓冲底层调用不带缓冲的系统调用。

8. 代码片段示例

```c
// 带缓冲：高效，但需注意刷新
FILE *fp = fopen("file.txt", "w");
fprintf(fp, "Hello");    // 可能还在缓冲区
fflush(fp);              // 强制刷新到内核
fclose(fp);              // 自动刷新

// 不带缓冲：直接调用，实时
int fd = open("file.txt", O_WRONLY | O_CREAT, 0644);
write(fd, "Hello", 5);   // 立即进入内核（但仍可能在内核页缓存）
close(fd);
```

#### 1. 命名管道

`fifo`的 `open` 是会阻塞的，必须要求要进行有读有写才能实现正常的打开<br>
管道文件只能通过 `mkfifo` 才能创建，创建完后是需要配置权限，并且 `open`<br>
匿名管道和命名管道唯一区别在创建方式不一样导致的可见性不一样。<br>
当多个进程同时向同一个管道的写端（fd[1]）写入数据时，内核需要处理这些并发请求。<br>
匿名管道和命名管道在内核中都是环形缓冲区，维护以下关键属性：
- 读偏移和写偏移（无锁或自旋锁保护）
- 等待队列（用于阻塞）
- 容量（默认 64KB，可调）
- PIPE_BUF（通常 4096 字节），定义原子写入的最大长度
- 多个进程可以通过共享同一文件表项（通过 fork、dup 或 open FIFO）获得同一个管道的多个读端或多个写端。

>多 reader 抢数据：谁读到不确定<br>
>多 writer 写乱序(大于PIPE_BUF):消息被混乱<br>
>多 writer 写乱序(小于PIPE_BUF):多次写会被一次读取<br>
>阻塞死锁<br>

使用场景 多生产者单消费者，且每个消息 ≤ `PIPE_BUF`
- 多个线程/进程写入一个管道，消费者读取（如日志聚合器）。
- 利用原子写入保证消息不交叉，且实现比消息队列更轻量（无需额外库）。

注意事项：
- 当管道读端已关闭，写端仍尝试 write 时，内核会向写进程发送 SIGPIPE 信号（默认终止进程）。signal(SIGPIPE, SIG_IGN) 忽略，或者检查 write 返回 -1 且 errno == EPIPE 时优雅处理。
- 父子进程互相等待对方数据，但双方都先写后读且管道容量有限，可能死锁。设计好读写顺序，或使用非阻塞模式 + select/poll。
- 通常应避免多读者，除非故意做负载均衡（任务队列）。
- 默认管道容量约 64KB（Linux），当写入超过剩余空间时，write 会阻塞直到有人读取。
- 小于4k的数据，pipe比uds快的多
