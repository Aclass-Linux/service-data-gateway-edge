#### I. 协议校验性能模型分析

**结论**：在 epoll/libuv 回调里直接做协议合法性校验（含 CRC-16），在一帧 256 字节极限包下耗时约 **1 微秒**，只占事件循环优秀预算 100 μs 的 1%。

**三条纪律**：
- ❌ 回调内 `printf` → 几十 ms → 改计数器 `atomic_fetch_add`
- ❌ CRC 表 Cache Miss → 25 μs → 512B 小表常驻 L1
- ❌ 回调内加锁 → 阻塞不确定 → 回调层零锁

#### II. 点表格式演变路径

| 阶段 | 方案 | 理由 | 最终状态 |
|------|------|------|---------|
| 起点 | `.bin` + `mmap` 零拷贝 | 性能最优 | ❌ 已移除 |
| 第1转 | 换成 JSON | 统一 config.json 工具链 | ❌ 内存问题 |
| 第2转 | XML | 流式解析大文件 | ❌ 依赖重 |
| 第3转 | SQLite 嵌入式 | 稳定可靠、ALTER TABLE 迁移 | ✅ 确定为配置库 |
| 细化 | SQLite 做配置库，纯内存做运行时 | 配置走磁盘，读写走内存 | ✅ 当前实现 |

#### III. 最终数据架构

```
SQLite config.db          // 配置库：southbound / northbound / route 三张表
  │ egw_ptable_open()
  ├─ SELECT * FROM ... ORDER BY device_id, sig_id
  ├─ qsort() 再确保有序
  ├─ sqlite3_close()     // 关闭连接，后续不碰磁盘
  └─ 返回 egw_ptable_t { 三个有序数组 }
        │
        └─ egw_ptable_sb_lookup(device_id, sig_id)
              └─ bsearch() 二分查找 O(log n)
```

### 遇到的问题

1. **sqlite 子模块文件删光了**：shell `for f in * .*` 的 glob 把 `.git` 和 `sqlite3.c` 全删了 → `git submodule deinit + update` 恢复，第二次用显式 `rm -rf` 逐个删除。

2. **sqlite 开发分支没有 amalgamation**：`./configure --dev && make sqlite3.c` 生成，依赖 tclsh 和 jimsh（集成在仓库内），过程顺利。

3. **加载器 `COUNT(*)` 调两次**：第一次 `SELECT COUNT(*)` 算大小 + 分配内存，第二次 `SELECT *` 填充数组。不如单次 `realloc` 循环省代码，但让加载阶段意图清晰。

### 关键设计决策记录

| 决策 | 选择 | 备选方案 | 原因 |
|------|------|---------|------|
| 运行时格式 | SQLite config + 内存数组 | `.bin` / JSON / XML / 纯 SQLite 运行时查询 | 磁盘可靠 + 内存快速 |
| 配置文件数 | 单文件 `config.db` 三表 | 每类点表独立文件 | 原子替换，简化管理 |
| 运行时值持久化 | 移除（不持久化） | mmap persist / SQLite runtime db | 简化，当前不需要 |
| 查找算法 | `bsearch()` O(log n) | 完美哈希 O(1) / SQLite 查询 | 点表规模小，bsearch 足够简单 |
| SQLite 编译选项 | `THREADSAFE=0` | 默认 THREADSAFE=1 | 单线程，省锁开销 |
| 点表 struct | `__attribute__((packed))` 删除 | 保持 packed | SQLite 的列自然处理对齐 |
