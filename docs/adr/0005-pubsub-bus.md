# ADR 0005: 南向北向通过内部发布订阅总线解耦

## 状态

已接受

## 背景

网关需要连接南向设备（Modbus RTU）和北向云平台（MQTT），两端之间存在多对多关系：
- 一个南向测点可能被多个北向通道消费（MQTT 上报、SQLite 缓存、Lua 脚本监听）
- 一个北向通道可能订阅多个南向设备的测点
- 命令下发同样需要从北向路由到南向

如果南向和北向直接耦合（南向采集函数直接调用北向上报函数），每增加一个消费者都需要修改生产者代码，违反开闭原则。

## 决策

在**任务线程内部**引入发布订阅总线。南向 Modbus Master 是生产者，采集到数据后以 `(device_id, sig_id, value)` 发布；北向 MQTT Reporter、SQLite 缓存、Lua 脚本引擎作为消费者订阅相关测点。

订阅粒度为 `(device_id, sig_id)` 复合键。**sig_id 是设备内唯一标识**，不同南向设备可以有相同的 sig_id，因此必须用 `(device_id, sig_id)` 才能唯一标识一个测点。订阅者明确订阅哪些 `(device_id, sig_id)` 对。

第一版只考虑单线程：总线不做线程安全设计，发布时在线程内同步完成高效分发。`egw_bus_publish()` 查订阅表后立即同步调用每个订阅者回调，直到所有回调返回后才返回。

订阅者回调必须是非阻塞的：不能等待 I/O，不能直接执行长耗时计算，不能在回调内跑完整脚本。需要异步执行的消费者（如 Lua 脚本、MQTT 发送、SQLite 写入）只能在回调内提交本线程事件循环任务或唤醒协程，然后立即返回。真正的 I/O 或脚本执行由事件循环后续调度。

未来多任务线程扩展时，每个线程拥有自己的线程内总线；跨线程发布通过 uv 异步投递到目标线程，再由目标线程自己的总线继续发布。

总线实现复用并扩展已有的 Channel 概念（双队列），新增 sig_id 注册和路由表。线程内生产者和消费者通过总线间接通信，互不感知对方存在。

路由表（`routing.bin`）是全局公用的只读对齐中心，不作为运行时热更新配置，由离线工具与南北向点表同源构建。南北向点表承载协议解析/编码以及**本侧的转换系数（scale/offset）和死区（deadband）**，遵循「谁用谁存」；路由表只承载 `(device_id, sig_id)` 的对齐契约：

1. **索引**：`(device_id, sig_id) → 南/北向点表条目`、北向 `(功能码, 地址) → (device_id, sig_id)`，全程 O(1)。
2. **双向地址映射**：`(device_id, sig_id) → 南向设备寄存器地址`、`北向从机寄存器地址 → (device_id, sig_id)`，服务南向上行与北向命令下发。
3. **核心数据类型**：每个 `(device_id, sig_id)` 在总线上流通时约定的类型，供收发双方解释裸值。

路由表**不存**转换系数和死区。总线消息是核心类型的裸值、不带类型标识，发布者按本侧系数转成核心类型后发布，消费者按 `(device_id, sig_id)` 查路由表得类型、再用本侧系数和死区处理。查找采用完美哈希或等价 O(1) 索引结构，运行时通过 `mmap` 只读访问。

## 否决的方案

**直接函数调用**：采集完成后直接调 `mqtt_publish()`、`sqlite_insert()` 等。简单但每增加消费者都需改采集代码，且无法支持热更新订阅关系。

**全局回调注册表**：生产者维护一个回调函数列表，消费者注册回调。比直接调用解耦，但无 Topic 过滤，所有消费者收到所有数据，需各自判断是否关心，浪费 CPU。

**事件队列式派发（所有消息先入队，稍后统一派发）**：语义统一，天然适合异步消费者，但会把第一版所有消息都变成异步语义，提前引入队列生命周期、背压、溢出处理和调试复杂度。第一版单线程优先选择同步回调，异步消费者自行投递任务。

**ZeroMQ / nanomsg**：成熟的进程内消息队列，功能强大但引入额外依赖。当前数据量（每个串口链路每秒几十条 Modbus 帧）不需要独立消息队列引擎的开销。

## 理由

- **生产者消费者解耦**：新增北向通道（如 WebSocket、HTTP 上报）无需修改南向代码
- **sig_id 精确路由**：消费者只接收订阅的测点，O(1) 哈希查找，避免无关数据遍历
- **总线值物理表示**：`egw_value_t` union（8 字节，无 tag），与 ADR-0008 的 `_Atomic uint64_t` 持久化槽位对齐。参见 ADR-0009。
- **统一数据流**：上行采集和下行命令通过同一条总线，对称设计降低复杂度
- **路由索引稳定**：路由表作为全局只读快速索引，运行时不修改，避免订阅关系热更新引入一致性和并发问题
- **基于现有 Channel 概念**：不是全新概念，是对已有双队列模型的自然扩展
