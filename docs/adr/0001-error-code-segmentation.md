# ADR 0001: 错误码方案

## 状态

已接受（2026-06-10 修订：去除模块前缀，改为按错误性质命名）

## 背景

项目需要一个统一的故障返回机制，替代现有模块中 NULL/default 的错误处理方式。

## 决策

1. 使用 `typedef int32_t egw_err_t` + `#define` 定义错误码常量，不使用 enum（负值 enum 常量可能触发 `-Wpedantic` 警告）
2. 顺序编号（`-1, -2, -3...`），新增在末尾追加，保持现有值不变
3. **错误码按错误性质命名，不体现模块归属**（如 `EGW_ERR_NOTFOUND` 描述"不存在"，不关心是文件还是 key）。模块来源通过函数名追踪
4. 所有错误码全局统一定义在 `egw_defs.h`，不由各模块自行定义
5. 不与 POSIX errno 混用，保持独立值域

## 理由

- **`#define` 优于 enum**：错误码需要负值，C11 enum 对负值常量的支持依赖实现，可能触发编译器警告
- **按性质命名优于按模块命名**：`EGW_ERR_OPEN` 可被任何模块复用（serial/MQTT/TCP），避免每个模块定义一套重复的打开失败码。调试时函数名已指示错误来源
- **全局定义优于模块分散定义**：错误码是跨模块的公共语言，集中定义保证编号唯一，避免冲突
- **不自建 errno 系统**：本项目的函数以返回成功/失败为主语义，业务数据通过输出参数传出，不适合 errno 的"全局变量 + 信号值"模式
