# ADR 0010: 所有功能保持可线程化扩展

## 状态

已接受（约束性原则，配合 ADR-0007）

## 背景

ADR-0007 确立了第一版单线程 + 未来多线程扩展的路径。但除了显式的多任务线程场景，还有一类隐式的线程化需求：**某些功能的执行可能需要移到独立线程，避免阻塞主事件循环**。

典型场景：
- **重计算脚本**：Lua 脚本执行复杂算法（如 FFT、数据拟合、批量统计），耗时数百毫秒到数秒，会阻塞主线程的采集和上报
- **大批量数据处理**：导出历史数据、生成报表、批量校验点表
- **阻塞性第三方库**：某些 C 库没有异步接口，调用会阻塞（如某些加密库、压缩库）

第一版虽然只实现单线程，但**代码设计不能引入线程化障碍**——如果某个模块使用了线程不安全的全局状态、或假设所有代码运行在同一线程，未来线程化时会被迫大范围重构。

## 决策

### 0. 线程分类

项目中的线程分为两类：

**主线程（任务线程）**：
- 持有完整的 `egw_runtime_t`（`egw_loop_t`、`egw_bus_t`、`egw_ptable_t`、`lua_State`）
- 处理 I/O、事件循环、数据分发
- 第一版只有一个主线程；未来可能有多个任务线程（如多串口链路隔离）

**单一功能线程**：
- 不持有完整 runtime，只有必要的通信手段和专属资源
- 典型例子：SQLite 写线程（只有 `sqlite3*` + `uv_async_t` + 任务队列）、ADR-0008 持久化线程、Lua 重计算线程池（只有 `lua_State` + `uv_async_t` + 只读 `egw_ptable_t` 视图）
- 通过 `uv_async_send` 与主线程通信，不参与数据分发

以下约束主要针对主线程，但单一功能线程同样禁止使用线程不安全的全局状态。

### 1. 禁止使用线程不安全的全局可变状态

任何模块都不应依赖进程级全局可变状态（`static` 全局变量持有运行时数据、`errno`、`strtok` 等非可重入函数）。

运行时状态必须归属 `egw_runtime_t` 或模块句柄，通过线程内单例 `egw_runtime_current()` 或显式句柄传递访问。

**允许的全局状态**：
- 只读配置（启动时初始化后不变）
- 常量表（错误码字符串、类型名映射）
- 线程安全的单例（用 `pthread_once` 初始化）

**禁止的全局状态**：
- 文件级 `static` 变量持有连接池、缓存、计数器等运行时数据
- 假设单线程的静态缓冲区（如 `static char buf[256]`）

### 2. 模块 API 必须可重入（线程安全或线程隔离）

每个模块的公共 API 必须选择以下两种线程模型之一：

**模型 A：线程隔离**（推荐，第一版采用）
- 每个线程持有独立的模块实例句柄（如 `egw_bus_t`、`lua_State`）
- 不跨线程共享句柄，API 不需要加锁
- 跨线程通信通过消息传递（`uv_async_send`）

**模型 B：线程安全**（仅用于确实需要跨线程共享的资源）
- 使用原子操作或互斥锁保护共享状态
- 必须在头文件注释中标注 `@threadsafe` 和锁粒度
- 例如：只读点表 mmap 映射可以跨线程共享（映射本身线程安全，写权限由离线工具独占）

**默认原则**：优先选择模型 A；只有在性能分析证明共享资源更优时才选模型 B。

### 3. 重计算功能必须可移出主线程

任何**预期执行时间 >10ms** 的功能都必须设计为可移到独立线程执行，包括但不限于：

- Lua 重计算脚本
- SQLite 批量查询/导出
- 加密/解密/压缩
- 大批量点表校验

实现路径：
1. 将计算逻辑封装为**纯函数**（输入 → 输出，无副作用）
2. 主线程准备输入参数、发送到工作线程（通过 `uv_async_send` + 队列）
3. 工作线程执行计算，完成后通过 `uv_async_send` 返回结果
4. 主线程在事件循环 tick 中接收结果、继续处理

第一版可以在主线程直接调用（假装同步），但 API 签名和数据流必须为异步做好准备。

### 4. 阻塞 I/O 必须隔离到独立线程

任何阻塞 I/O（`pwrite`、`fdatasync`、`sqlite3_step`）都不能在主事件循环线程执行。

已实施的例子：
- ADR-0008 运行时值持久化：独立持久化线程做 `pwrite`/`fdatasync`
- SQLite 时序缓存：应使用独立写线程或 `uv_queue_work` 包装

### 5. 生命周期管理：引用计数或所有权转移

跨线程传递数据时，内存所有权必须清晰：

**方案 A：所有权转移**
- 发送方分配内存、填充数据、转移所有权给接收方
- 接收方处理完成后负责释放
- 适用于一次性传递的大块数据

**方案 B：引用计数**
- 使用原子引用计数（`_Atomic int ref_count`）
- 发送方和接收方各持有一个引用，最后释放者释放内存
- 适用于需要在多个线程保留副本的场景（如 Packet 对象池）

第一版单线程可以跳过引用计数，但数据结构必须预留 `ref_count` 字段。

## 否决的方案

**第一版就实现所有功能的多线程版本**：提前优化，浪费开发资源，且没有性能数据支撑。正确路径是先实现单线程版本、测量瓶颈、再针对性线程化。

**允许模块使用线程局部存储（TLS）缓存状态**：TLS 隐藏了线程依赖，调试困难。优先使用显式句柄传递或 `egw_runtime_current()`。

**第三方库可以使用线��不安全 API**：错误。如果第三方库不是线程安全的（如某些 Lua C 模块），必须确保每个线程持有独立实例，或在调用前加锁保护。

## 理由

- **避免技术债务**：第一版就遵循线程化约束，未来扩展时不需要大范围重构
- **性能瓶颈可定向优化**：重计算脚本、SQLite 批量查询等明确的阻塞点可以低成本移到工作线程
- **保持代码简洁**：第一版不实现线程池，只是设计时不引入障碍；API 和数据结构干净，后续加线程不会破坏现有调用方
- **测试友好**：线程隔离模型让单元测试不需要处理并发竞态

## 后果

- 代码审查时需检查：是否使用了线程不安全的全局状态、`strtok` 等非可重入函数
- 模块 API 设计时需明确线程模型（隔离 or 安全），并在头文件注释
- 重计算功能即使第一版在主线程执行，API 也要设计为异步风格（回调或 future）
- 第一版不需要实现线程池，但数据结构要预留引用计数字段、API 要支持异步返回

## 实施检查清单

新增模块时确认：
- [ ] 无文件级 `static` 全局变量持有运行时数据
- [ ] 不使用 `strtok`、`strerror`、`localtime` 等非可重入函数（改用 `_r` 版本）
- [ ] 模块句柄通过 `egw_runtime_t` 或显式参数传递，不通过全局变量
- [ ] 预期执行时间 >10ms 的功能设计为可移出主线程（纯函数 + 异步 API）
- [ ] 跨线程传递的数据结构有清晰的所有权管理（转移 or 引用计数）
