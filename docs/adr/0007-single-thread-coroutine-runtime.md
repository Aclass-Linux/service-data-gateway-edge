# ADR 0007: 线程内事件驱动运行时 + uv 跨线程投递

## 状态

已接受（细化 ADR 0004 的线程模型，与 ADR 0005、ADR 0006 协同）

## 背景

网关的并发模型有两个相互关联的决策点：

1. **线程内执行模型**：南向 Modbus 采集、北向上报、Pub/Sub 分发、轻量脚本协程都应在同一个任务线程内部以事件驱动方式运行。线程内消息发布优先考虑高效率，不为未来跨线程场景提前引入锁和队列开销。

2. **跨线程扩展模型**：未来可能引入多个任务线程或线程池。每个任务线程内部仍保持同样的事件驱动结构；线程之间不共享可变运行时状态，跨线程消息发布通过 libuv 的异步投递能力完成。

3. **状态所有权**：当前 App 层用文件级全局/静态变量持有运行时状态（`gateway_app.c` 中的 `g_inst`、`port_params[]`、`port_tps[]`、`port_count`），散落各处、无法支持多个任务线程。

第一版目标是单线程运行，不实现多任务线程和跨线程消息路由；但 API 和术语不能阻断后续扩展。

## 决策

### 1. 第一版只实现单线程

第一版只有一个任务线程、一个 `egw_runtime_t`、一个 `egw_loop_t`、一条线程内 Pub/Sub 总线。所有 I/O、定时器、协议处理、总线分发和轻量 Lua 协程都在这个线程内执行。

线程内不引入锁；总线发布以高效同步分发为目标。

### 2. 每个任务线程内部都是一致的事件驱动运行时

未来引入多个任务线程时，每个线程都持有自己的线程内运行时：

```
egw_runtime_t {
    egw_loop_t   *loop;     /* ADR 0006，线程内事件循环 */
    egw_bus_t    *bus;      /* ADR 0005，线程内 Pub/Sub 总线 */
    egw_ptable_t *ptable;   /* ADR 0003，只读点表视图 */
    lua_State    *L;        /* 本线程脚本状态 */
    /* SQLite 缓存、MQTT Reporter 等后续按线程职责挂载 */
}
```

`egw_runtime_t` 是**线程内单例**：每个任务线程一份，通过 thread-local 的 `egw_runtime_current()` 访问，不跨线程共享，也不使用进程级全局单例。

每个任务线程内部的编程模型保持一致：事件循环驱动 I/O 和定时器，总线在线程内同步分发消息，Lua 协程在线程内 yield/resume。

### 3. 跨线程消息通过 uv 异步投递

线程内发布走本线程总线；跨线程发布不直接调用目标线程的总线，也不共享目标线程 runtime。

跨线程消息统一通过 libuv 的异步投递机制（如 `uv_async_send`）唤醒目标线程事件循环，由目标线程在自己的 loop tick 内接收消息，并在本线程总线上继续发布。

因此跨线程边界是：

```
源线程 bus -> 跨线程投递队列/uv_async -> 目标线程 loop -> 目标线程 bus
```

第一版不实现该跨线程桥，只保留模型边界。

### 4. 重计算脚本的扩展位置

ADR 0004 中的重计算脚本可以在未来调度到**单一功能线程池**中执行。这些工作线程只持有独立 `lua_State`、只读的 `egw_ptable_t` 视图、`uv_async_t` 通信句柄和任务队列，**不需要完整 runtime**（无 `egw_bus_t`/`egw_loop_t`，因为不参与数据分发和 I/O 处理）。主线程通过 `uv_async_send` 提交计算任务，工作线程完成后返回结果。

第一版不实现线程池；重计算脚本能力作为后续扩展。

`egw_runtime_t` 专属于需要处理 I/O 和数据分发的主线程，只持有子系统句柄和生命周期，不承载业务状态、不写业务逻辑，以免退化为上帝对象。单一功能线程（SQLite 写线程、持久化线程、Lua 计算线程）不持有完整 runtime，只有必要的通信手段和专属资源句柄。

## 否决的方案

**第一版就实现多线程总线**：提前引入锁、跨线程订阅关系、消息顺序规则和对象生命周期管理，会拖慢当前单线程主链路，且还没有足够业务压力证明需要。

**进程级全局 runtime**：最省事，但一旦引入多个任务线程，`lua_State`、loop、bus 的归属会混乱，无法表达线程隔离。

**所有 publish 都走异步队列**：模型统一，但牺牲线程内发布效率。当前第一版只有单线程，线程内直接同步分发更合适。

**显式句柄层层透传（每 API 第一参数传 `egw_runtime_t *`）**：依赖在签名上最直观，但调用链冗长。本项目采用线程内单例语义，跨层代码通过 `egw_runtime_current()` 获取本线程上下文。

## 理由

- **第一版高效**：单线程内无锁、同步分发，避免为暂未实现的多线程付运行时成本。
- **未来一致**：多任务线程不是另一套模型；每个线程内部仍是事件驱动 runtime，只在线程边界增加 uv 投递桥。
- **线程隔离清楚**：`egw_runtime_t`、`egw_loop_t`、`egw_bus_t`、`lua_State` 都归属某个任务线程，不跨线程直接访问。
- **与 ADR 0006 一致**：Core 层封装事件循环；runtime 是 loop 之上聚合 bus、点表、脚本状态的线程内上下文。

## 后果

- App 层需重构：移除 `g_inst`、`port_*[]`，状态收归线程内 `egw_runtime_t`。
- 需实现 thread-local 运行时存取（`egw_runtime_current()` / 初始化绑定）。
- 第一版总线不需要线程安全；只保证线程内同步分发。
- 后续多线程扩展需新增跨线程投递桥，但不能让线程 A 直接访问线程 B 的 runtime 或 bus。
