## 2、通讯模型

### 2.1 定义

1. 主进程是数据处理进程，所有的其他进程通讯都和主进程进行通讯，称之为Server进程
2. 采集进程和转发进程都是和数据处理进程进行处理，称之为client进程

```
    采集进程             处理进程               转发进程

    发起请求    ----->   处理请求

    mmap本地   <-----    创建共享内存

```

如上图所示，Client 进程和 Server 进程第一次通讯时需要建立 TCP 连接或是 Unix Domain Socket 连接，然后通过该连接完成协议初始化。初始化涉及共享内存的初始化和映射等过程，初始化完成之后可以通过共享内存进行全双工通讯。

## 3、通讯协议

通讯协议定义了UDS连接所需要的消息格式

### 3.1 消息格式

Client和Server进程在连接上所传递的一条消息如下：

```c
struct {
    uint8_t magic;              // Magic number 用于标识协议本身，有类型的定义。
    uint8_t header_payload[];   // 根据magic头解析活动头的内容
}
```

首先由一个 magic 的头进行定义，有不同的head和payload解决方案

| magic头 | 解释 |
|:-------:|:-------|
|   0xA1  | 单个的设置、查询消息，顺序模式，总长度不过256， 用换行结尾不用CRC| 
|   0xA3  | 多参数设置，使用共享内存进行，有前期握手，然后再通过共享内存完成 |
|

## 4、共享内存设计

### 4.1 共享内存初始化流程

1. Client 先建立 TCP 或 UDS 连接，作为控制面连接，后续用于握手、同步通知和异常回退。
2. Client 发送 `0x20` 协议初始化帧，请求进入共享内存模式。
3. 数据处理进程作为 Server，负责决定本次连接使用的共享内存总大小、内部布局和各功能区偏移，并把这些完整信息写入 `SharedMemoryHeader`。
4. Server 回复 `0x21` 协议初始化回复。
`0x00` 表示通过 `FD` 方式传递共享内存对象。
`0x01` 表示通过共享内存名称或文件路径方式建立映射。
`0x02` 表示初始化失败，连接保持控制面能力，但不进入共享内存模式。
5. 当使用 `FD` 方式时，Server 通过 `0x21` 表示已准备接收 `FD`，再由 Client 通过 `UDS` 附加 `FD` 传送共享内存对象。
6. Client 完成 `mmap` 后，不直接进入收发流程，而是先读取 `SharedMemoryHeader`，校验版本、总大小、队列布局、槽池规格等元数据。
7. Client 完成 Header 校验后回复 `0x22`，表示共享内存映射和布局解析完成。
8. 后续双方通过 `0x23` 进行同步通知，实际业务数据写入共享内存，不再直接走连接。

### 4.2 共享内存映射方式

| 方式 | 说明 | 适用场景 |
| :-- | :-- | :-- |
| `memfd` | 由 Client 创建匿名共享内存，经过 `UDS` 传递 `FD`，生命周期跟随进程，退出后由系统回收 | Linux 3.17+，推荐默认方式 |
| 文件路径 | 由 Client 创建共享内存文件并传递名称或路径，Server 再按名称映射 | 无法传递 `FD` 或需要跨进程组定位共享内存时 |

设计约束：

1. `TCP` 连接不支持 `FD` 传递，因此使用 `memfd` 时控制连接必须是 `UDS`。
2. 文件路径方式虽然兼容性更高，但异常退出时通常需要额外清理逻辑。
3. `memfd` 方式更适合无状态进程，清理成本更低。

### 4.3 共享内存布局设计

共享内存布局由数据处理进程在初始化阶段统一决定，不在协议中写死。处理进程完成布局规划后，将完整结果写入 `SharedMemoryHeader`，Client 在 `mmap` 后按 Header 解析内部结构。

`0xA3` 模式下，共享内存建议划分为 3 个逻辑区域：

1. `SharedMemoryHeader`：保存版本号、总大小、队列偏移、槽池规格等完整布局元数据。
2. `SlotPool`：按固定规格维护消息槽位，用于存放实际业务数据。
3. `IO Queue`：描述“有哪些消息已经写入共享内存，等待对端处理”。

推荐布局：

| 区域 | 作用 |
| :-- | :-- |
| 头部区 | 保存 `SharedMemoryHeader`，记录版本号、总大小、队列偏移、槽池定义等元数据 |
| 槽池管理区 | 保存多个槽池定义及其空闲管理信息，每个槽池对应一种固定块大小 |
| 数据区 | 保存实际消息槽位，承载业务数据 |
| 事件队列区 | 保存双向 `IO Queue` 和 `Event` 数组，用于跨进程通知收割 |

共享内存布局示意图：

```text
+----------------------------------------------------------------------------------+
|                              Shared Memory Total Area                            |
+----------------------------------------------------------------------------------+
| SharedMemoryHeader                                                               |
| magic | version | totalSize | headerSize | c2sQueueOffset | s2cQueueOffset ...  |
+----------------------------------------------------------------------------------+
| SlotPool Meta Area                                                               |
| Pool-0 Meta | Pool-1 Meta | Pool-2 Meta | ...                                    |
+----------------------------------------------------------------------------------+
| SlotPool Data Area                                                               |
| Pool-0 Slots (256B) | Pool-1 Slots (1KB) | Pool-2 Slots (2KB) | ...              |
+----------------------------------------------------------------------------------+
| Event Queue Area                                                                 |
| C2S IO Queue | S2C IO Queue | Event Array / Reserve                              |
+----------------------------------------------------------------------------------+
```

### 4.4 SharedMemoryHeader / SlotPool 设计

共享内存采用“处理进程生成布局 + Header 发布元数据 + 多规格固定槽池管理”的方式，兼顾性能、可维护性和扩展性。

`SharedMemoryHeader` 推荐字段：

| 字段 | 说明 |
| :-- | :-- |
| magic | Header 固定标识，用于确认这是合法的共享内存头 |
| version | 共享内存布局版本号 |
| totalSize | 共享内存总大小 |
| headerSize | Header 自身大小 |
| c2sQueueOffset | Client 到 Server 队列偏移 |
| s2cQueueOffset | Server 到 Client 队列偏移 |
| slotPoolCount | 当前槽池数量 |
| featureFlags | 布局特性标志位 |

`SlotPool` 推荐字段：

| 字段 | 说明 |
| :-- | :-- |
| slotSize | 当前槽池单个槽位大小，例如 `256B/1KB/2KB` |
| slotCount | 当前槽池槽位数量 |
| slotAreaOffset | 当前槽池数据区起始偏移 |
| freeQueueOffset | 当前槽池空闲管理区偏移 |
| freeCount | 当前空闲槽位数量 |
| allocCount | 分配计数，用于监控 |
| freeOpCount | 回收计数，用于监控 |

分配回收原则：

1. 发送方优先选择“刚好能容纳数据”的最小槽位规格，减少内存浪费。
2. 单条消息默认只占用一个槽位，不建议在第一版设计中引入多槽拼接。
3. 分配时从对应槽池的空闲管理区获取可用槽位，回收时再放回空闲管理区。
4. 槽池的空闲计数、分配计数和回收计数修改都必须使用原子操作。
5. 若消息超过最大槽位规格，建议直接回退到连接发送，而不是强行拼接多个槽位。

### 4.5 SharedMemoryHeader 解析规则

Client 在完成 `mmap` 后，应按以下顺序解析 `SharedMemoryHeader`：

1. 先读取固定头部，至少包含 `magic`、`version`、`headerSize` 和 `totalSize`。
2. 校验 `magic` 是否正确，若不是合法标识，则本次共享内存初始化失败。
3. 根据 `version` 选择对应的 Header 解析方式，若版本不支持，则不得进入共享内存模式。
4. 校验 `headerSize` 是否合理，且不得超过 `totalSize`。
5. 根据 Header 中的队列偏移、槽池数量和槽池定义，继续解析完整布局信息。
6. 校验所有偏移和长度是否落在共享内存有效范围内，防止越界访问。
7. 完成全部校验后，Client 才能回复 `0x22`，表示共享内存布局解析成功。

### 4.6 消息槽位设计

| 字段 | 说明 |
| :-- | :-- |
| data | 实际数据内容 |
| capacity | 当前槽位最大承载字节数 |
| dataLen | 当前消息有效数据长度 |
| msgType | 当前消息类型 |
| flags | 标志位，至少包含“是否已分配”“是否已写完”等状态 |
| reserve | 预留字段，便于后续扩展 |

说明：

1. 对端只需要根据 `Event` 中记录的槽位编号或偏移，就能读取完整消息。
2. 由于单条消息默认只占一个槽位，读取路径更简单，维护成本也更低。

### 4.7 IO Queue 与同步机制

连接只负责“通知”，数据本体放在共享内存中。推荐为每个方向各维护一个 `IO Queue`：

1. Client `SendQueue` = Server `RecvQueue`
2. Server `SendQueue` = Client `RecvQueue`

`IO Queue` 推荐字段：

| 字段 | 说明 |
| :-- | :-- |
| capacity | 当前队列可容纳的事件数 |
| head | 读指针，单调递增 |
| tail | 写指针，单调递增 |
| flags | 队列标志位，至少包含 `workingFlag` |

`Event` 推荐字段：

| 字段 | 说明 |
| :-- | :-- |
| slotOffset | 消息槽位在共享内存中的偏移，或槽位编号 |
| dataLen | 当前消息有效长度 |
| reserve | 预留 `8byte`，可用于业务扩展，如虚拟连接ID、状态位等 |

单次通讯流程：

1. 发送方从合适规格的 `SlotPool` 中申请一个槽位，并把业务数据写入该槽位。
2. 发送方把槽位偏移或槽位编号封装为一个 `Event`，写入 `IO Queue`。
3. 发送方将 `workingFlag` 置为 `1`，并发送 `0x23` 通知对端。
4. 接收方收到 `0x23` 后，从 `IO Queue` 批量取出待处理事件。
5. 接收方根据 `Event` 中的槽位信息读取完整数据。
6. 数据处理结束后回收对应槽位到原槽池。
7. 队列清空后，接收方将 `workingFlag` 复位为 `0`。

批量收割策略：

1. 当 `workingFlag=1` 时，后续发送方只写队列，不重复发送通知。
2. 接收方被唤醒一次后，可以批量处理多个事件，减少连接面通知开销。

### 4.8 共享内存内协议封装规则

对于写入共享内存的数据内容，建议保持当前定义的 `0xA3` 业务帧格式不变，但增加以下约束：

1. 共享内存中的一条完整业务记录，以 `固定帧头 + 长度 + 业务头 + 数据 + 前四列重复 + 校验` 组成。
2. `前四列重复` 用于快速校验共享内存读写边界是否错位，建议重复 `固定帧头 + 长度` 的前 4 字节内容。
3. `数据长度` 表示共享内存内业务数据区的有效长度，不包含尾部校验。
4. 第一版设计中，建议一条共享内存记录对应一个槽位，避免引入多槽拼接复杂度。

### 4.9 异常回退设计

当共享内存资源不足时，不建议立即扩容，而应回退到控制面连接发送，避免系统复杂度失控。

回退策略：

1. 当无法从合适规格的 `SlotPool` 申请到可用槽位时，发送方将该次业务转为连接发送。
2. 当 `IO Queue` 已满时，说明接收方处理压力过高，此时发送方应阻塞等待或返回忙状态，不继续挤压共享内存。
3. 回退消息仍沿用 `0xA3` 语义，但通过连接直接发送完整数据，供接收方按相同业务逻辑处理。
4. 连接天然带缓冲和背压能力，可作为共享内存耗尽时的保护通道。

### 4.10 生命周期与清理策略

| 场景 | 推荐策略 |
| :-- | :-- |
| `memfd` 模式 | 依赖进程生命周期，进程退出后由系统自动回收 |
| 文件路径模式 | 由存活进程或守护进程负责 `unmap` 和文件清理 |
| 对端异常断开 | 另一侧收到连接关闭事件后，主动解除映射并停止收发 |

建议：

1. `memfd` 作为默认实现路径。
2. 文件路径模式仅作为兼容或调试方案。
3. 共享内存布局中保留版本字段，后续即使调整 `SharedMemoryHeader`、`SlotPool` 或 `Event` 结构，也能通过版本协商兼容升级。
