# 各阶段详细目标计划

## M1：单进程版网关（第1-6周）

### 总体目标
实现一个基于C语言的单进程网关原型，完成模拟数据采集、MQTT上传、守护进程化和基本可靠性保障。

### 详细周计划

| 周次 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第1周** | 文件IO、FIFO、信号 | 1. 实现模拟数据源程序 `simulator.c`<br>2. 每秒生成随机温度值（20-100）<br>3. 写入命名管道 `/tmp/temp_fifo`<br>4. 处理SIGPIPE信号（防止写端无读端时崩溃） | `open`, `mkfifo`, `write`, `sleep`, `signal` | `simulator` 可执行程序 | 运行 `./simulator` 后，`cat /tmp/temp_fifo` 能看到持续输出的温度数据 |
| **第2周** | 进程、管道读取 | 1. 实现主控程序 `gateway_v1.c`<br>2. 打开FIFO读取数据<br>3. 打印到stdout（带时间戳）<br>4. 处理FIFO打开阻塞问题 | `fork`（可选），`read`, `time`, `printf` | `gateway_v1` 可读数据 | 两个终端分别运行 simulator 和 gateway_v1，能看到类似 `[2025-05-26 08:00:01] 温度=45.2` 的输出 |
| **第3周** | 守护进程、syslog | 1. 增加 `--daemon` 参数，调用 `daemon(0,0)`<br>2. 将 `printf` 替换为 `syslog` 记录日志<br>3. 增加 `-d` 参数（前台调试模式）<br>4. 实现信号处理（SIGTERM优雅退出） | `daemon`, `syslog`, `openlog`, `signal`, `getopt` | 可守护进程化的网关 | 运行 `./gateway_v1 --daemon` 后，进程在后台运行，日志写入 `/var/log/syslog`，`kill` 后正常退出 |
| **第4周** | TCP Socket、MQTT | 1. 集成 Paho MQTT C 客户端库<br>2. 连接公共 broker（如 `broker.emqx.io`）<br>3. 每收到一个温度值，发布到主题 `gw/temp`<br>4. 处理MQTT断连重试 | `socket`, `MQTTClient`, JSON 封装 | 可上传数据的网关 | 运行 `mosquitto_sub -h broker.emqx.io -t 'gw/temp'` 能收到温度数据 |
| **第5周** | 看门狗、子进程管理 | 1. 主控进程 fork 出采集子进程（替代原FIFO方式）<br>2. 采集子进程负责生成数据并通过管道发送给主控<br>3. 主控监控子进程，若子进程异常退出则自动重启<br>4. 处理 SIGCHLD 信号 | `fork`, `pipe`, `waitpid`, `sigaction` | 高可用网关 | 手动 kill 采集子进程，主控能立即重启新子进程，数据不中断 |
| **第6周** | 调试与稳定性 | 1. 使用 `gdb` 调试段错误<br>2. 使用 `valgrind` 检查内存泄漏<br>3. 编写 Makefile，支持 `make` 和 `make clean`<br>4. 编写 README，包含编译、运行、测试步骤 | `gdb`, `valgrind`, `make` | 稳定版单进程网关 | 运行 24 小时无崩溃，无内存泄漏，代码仓库结构清晰 |

### M1 最终交付物
- 源代码：`simulator.c`, `gateway_v1.c`, `Makefile`
- 配置文件示例（如有）
- README.md：包含功能介绍、编译运行指南、依赖库安装（libpaho-mqtt3）
- 日志示例截图

## M2：C++多线程边缘处理引擎（第7-11周）

### 总体目标
用现代C++重写网关，引入多线程生产者-消费者模型，实现环形缓冲区和可配置的规则引擎。

### 详细周计划

| 周次 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第7周** | 类与RAII、智能指针 | 1. 创建 `DataPoint` 类（时间戳、值、质量戳）<br>2. 创建 `RingBuffer` 类（循环队列，线程安全）<br>3. 使用 `std::unique_ptr` 管理资源<br>4. 将原有 C 代码逐步改写为 C++ | 构造函数/析构函数，`std::mutex`, `std::unique_ptr` | C++ 版本基础框架 | 编译通过，运行单线程采集和上传功能与原版一致 |
| **第8周** | 生产者-消费者模型 | 1. 创建采集线程（生产者）：从 FIFO 或模拟源读数据，放入 `RingBuffer`<br>2. 创建处理线程（消费者）：从 `RingBuffer` 取出数据，执行规则检查并触发上传<br>3. 使用 `std::thread`, `std::condition_variable` 实现同步 | 线程安全队列，条件变量，`std::thread` | 多线程数据流 | 采集线程和处理线程并发运行，无数据竞争，吞吐量提升 |
| **第9周** | 线程池 | 1. 将单消费者扩展为线程池（3-5个工作线程）<br>2. 每个工作线程独立处理数据点<br>3. 实现线程池的优雅关闭 | `std::vector<std::thread>`, 任务队列 | 线程池模块 | 可动态调整线程数，处理速度随线程数增加而提升 |
| **第10周** | 模板与规则引擎 | 1. 实现 `Rule<T>` 模板类（支持不同数据类型）<br>2. 从配置文件（JSON或INI）加载规则，例如 `"temp > 80"`<br>3. 用 lambda 表达式实现规则函数 | 模板，`std::function`, `std::string` 解析 | 可配置规则引擎 | 修改配置文件中的阈值后，无需重启网关即可生效（通过信号触发重载） |
| **第11周** | 移动语义与异步MQTT | 1. 将 MQTT 模块封装为 C++ 类<br>2. 使用 `std::move` 传递数据，减少拷贝<br>3. 实现异步发布（非阻塞）<br>4. 增加自动重连机制 | 移动语义，`std::async` 或回调 | 高性能上传模块 | MQTT 连接断开后能自动重连，重连期间数据不丢失 |

### M2 最终交付物
- C++ 版本网关源码（`.cpp`, `.hpp`），使用 CMake 组织
- 配置文件示例（`rules.json`）
- 多线程吞吐量测试报告（对比单线程版本）
- 规则引擎演示（修改规则后动态生效）

## M3：远程管理能力（第12-14周）

### 总体目标
为网关增加 REST API 配置接口、持久化存储和断网续传能力。

### 详细周计划

| 周次 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第12周** | HTTP Server | 1. 集成 libmicrohttpd 库<br>2. 实现 GET `/config` 返回当前配置（JSON 格式）<br>3. 实现 POST `/config` 更新配置（接收 JSON 请求体）<br>4. 支持修改采集周期、报警阈值 | HTTP 协议，REST API，JSON 解析（如 nlohmann/json） | 配置服务模块 | 使用 `curl` 测试：`curl http://localhost:8080/config` 返回配置；`curl -X POST -d '{"threshold":85}'` 更新成功 |
| **第13周** | 配置持久化与热加载 | 1. 将配置保存到文件（如 `/etc/gateway/config.json`）<br>2. 网关启动时加载配置<br>3. 支持 SIGHUP 信号触发重载配置<br>4. 更新配置后，规则引擎动态生效 | 文件 I/O，`signal`，原子操作 | 配置管理模块 | 修改配置文件后，执行 `kill -HUP <pid>`，网关日志显示重载配置，新规则生效 |
| **第14周** | SQLite 断网续传 | 1. 集成 SQLite3 库<br>2. 创建本地数据表 `pending_data`（时间戳、主题、消息）<br>3. MQTT 连接正常时直接发布；断连时存入数据库<br>4. 增加后台线程定期检查 MQTT 状态，恢复后批量补传并删除已传记录 | SQLite3 API，定时器，事务 | 缓存与补传模块 | 断开网络，网关将数据写入 SQLite；重连后，数据自动补传，数据库清空 |

### M3 最终交付物
- REST API 文档（支持的操作和参数）
- 配置文件格式说明
- SQLite 数据库 schema
- 断网续传测试用例（断开/恢复网络脚本）

### 详细周计划

| 周次 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第15周** | 交叉编译 | 1. 安装交叉编译工具链（arm-linux-gnueabihf-g++）<br>2. 交叉编译依赖库（paho-mqtt, libmicrohttpd, sqlite3）<br>3. 修改 CMake 工具链文件，生成 ARM 可执行文件<br>4. 在 QEMU 用户态模拟或树莓派上测试 | 交叉编译，CMake toolchain，QEMU | ARM 版网关可执行文件 | 在 ARM 环境（或 QEMU）中运行网关，功能与 x86 一致 |

## M5：Rust 采集模块集成（第4-6个月，穿插）

### 总体目标
学习 Rust 基础，用 Rust 实现一个安全高效的采集模块，通过 FFI 与 C++ 主控通信。

### 详细月计划

| 月份 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第4个月** | Rust 语言基础 | 1. 安装 Rust 工具链<br>2. 通读《The Rust Book》第1-10章<br>3. 完成所有权、结构体、枚举、错误处理、泛型、trait 练习<br>4. 编写小项目：猜数字、命令行 grep | `cargo`, 所有权规则，`Result<T,E>` | Rust 语法笔记 | 能独立编写 Rust 程序处理文件读写和简单数据结构 |
| **第5个月** | FFI（Rust → C++） | 1. 学习 `extern "C"`、`#[no_mangle]`<br>2. 用 Rust 实现一个简单的 `add(a,b)` 函数，编译为静态库<br>3. 用 C++ 调用该函数<br>4. 编写 Rust 采集模块：读取模拟温度（或串口），返回结构体 | FFI，`std::ffi`，内存布局 | `libcollector.a` 及头文件 | C++ 主程序能成功调用 Rust 函数，获取温度值 |
| **第6个月** | Rust 重写环形缓冲区 | 1. 用 Rust 实现线程安全的 `RingBuffer<T>`<br>2. 实现 `Send` + `Sync` 保证线程安全<br>3. 编写 benchmark 对比 C++ 版本的性能<br>4. 将 Rust 模块集成到网关主控（替换原有 C++ 缓冲区） | 并发编程，`Mutex`，`benchmark` | Rust 版环形缓冲区 | Rust 版本性能不低于 C++ 版本，无内存不安全代码 |

### M5 最终交付物
- Rust 采集模块源码（`collector` crate）
- Rust 环形缓冲区 crate
- FFI 绑定代码与构建脚本（`build.rs`）
- 性能对比图表（C++ vs Rust）

### 详细月计划

| 月份 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第7个月** | ROS 2 基础 | 1. 安装 ROS 2 Humble<br>2. 学习节点、话题、服务、动作概念<br>3. 编写简单的 `talker`/`listener` 示例<br>4. 熟悉 `rclcpp`（C++客户端库） | ROS 2 图模型，`rclcpp::Node` | ROS 2 环境配置 | 能运行小乌龟仿真并控制 |
| **第8个月** | 网关 ROS 节点化 | 1. 将网关主控改造为 ROS 2 节点<br>2. 数据采集模块发布 `sensor_msgs::msg::Temperature` 话题<br>3. 规则引擎改为 ROS 服务（触发报警）<br>4. 用 `rclrs`（Rust 绑定）实现采集节点 | ROS 2 消息定义，服务，`rclrs` | 网关 ROS 节点 | 启动后，`ros2 topic list` 能看到温度话题，`ros2 topic echo` 能看到数据 |

## M7：Yocto 定制镜像集成网关（第10-12个月）

### 总体目标
使用 Yocto 为嵌入式 Linux 构建定制发行版，并将网关软件预集成到镜像中。

### 详细月计划

| 月份 | 学习主题 | 具体任务 | 技术要点 | 产出物 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **第10个月** | Yocto 基础 | 1. 安装 Yocto 环境（Poky）<br>2. 在 QEMU 上构建 `core-image-minimal` 并启动<br>3. 理解 recipe、layer、bbappend 文件<br>4. 编写一个简单的 helloworld recipe | BitBake，OpenEmbedded | 可运行的 QEMU 镜像 | `runqemu qemux86` 能启动 Linux 系统 |
| **第11个月** | 为网关创建 recipe | 1. 将网关源码放置为自己的 layer（`meta-industrial-gateway`）<br>2. 编写 `gateway.bb` recipe，从 Git 拉取代码并编译<br>3. 添加依赖：libpaho-mqtt, libmicrohttpd, sqlite3 等<br>4. 添加 systemd 服务文件，使网关开机自启 | 自定义 layer，recipe 语法，`SRC_URI`，`DEPENDS` | 包含网关的镜像 | 构建出的镜像烧录到 QEMU 或板卡后，`systemctl status gateway` 显示 active |
| **第12个月** | 镜像优化与测试 | 1. 修改 `local.conf` 优化 rootfs 大小（移除不必要包）<br>2. 启用 `rm_work` 节省构建空间<br>3. 测试网关在目标硬件上的长期稳定性<br>4. 编写 Yocto 构建文档 | 镜像定制，包选择 | 生产级嵌入式镜像 | 镜像大小不超过 256MB，网关在 QEMU 中稳定运行 72 小时 |

### M7 最终交付物
- Yocto layer `meta-industrial-gateway`
- Gateway recipe 和 systemd 服务 bbappend
- 构建文档（硬件要求、构建步骤、烧录方法）
- 最终镜像文件（`.wic` 或 `.img`）
